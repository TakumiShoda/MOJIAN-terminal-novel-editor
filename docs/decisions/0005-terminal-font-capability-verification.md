# ADR 0005：终端字体能力表尚未真机验证，以 `mj doctor` 为准

日期：2026-07-19
状态：已接受（含未完成项）

## 背景

doc.md §2.1 给了一张终端字体能力表（xterm 系走 OSC 50、kitty 仅字号且需远程控制、
Alacritty/WT 无），并把整张表标成 `[VERIFY]`，附了一句明确的禁令：

> `[VERIFY]` 各终端的实际能力必须在真机上验证，**不得照抄文档表格**。

M6 要实现 `FontController` 多后端，绕不开这张表。

## 问题

开发环境里没有 kitty、Alacritty、WezTerm、Windows Terminal 可供验证——
连 `TERM` 都是空的（`mj doctor` 在此环境如实报「未知终端」）。
也就是说：**我无法履行 `[VERIFY]` 的要求**。

有三条路：

1. 照抄文档表格，当作已验证 —— 直接违反禁令，且一旦文档有错，错误会被固化进
   代码，用户看到「可改字体」却改不动。
2. 什么都不做，等到能验证为止 —— M6 的外观一整块都得停。
3. 按文档实现，但**把「未验证」这件事本身做进产品里**。

选 3。

## 决定

**能力判定分成「先验」与「实测」两层，用户看到的一律是后者。**

- `TerminalKind::expected_caps()` 是先验，方法名里的 `expected` 就是提醒：
  它来自 §2.1 的表，未经真机验证。代码注释逐条写明哪些存疑
  （OSC 50 的空参数是否等于「恢复默认」尤其没把握）。
- `mj doctor` 是实测手段：它在**用户自己的终端里**跑，报告实际探测结果。
  §12.2 本来就说「doctor 是排查用户环境问题的第一手段，别省」——
  它同时也是这条 ADR 的兑现方式。
- 报告里凡是推断出来的，措辞明写「推断」，并点出前置条件
  （如 kitty 需 `allow_remote_control`，没开就是改不动）。不冒充事实。

**探测顺序上有一处不照抄文档的地方**：文档提到按 `TERM_PROGRAM / TERM /
KITTY_WINDOW_ID / WEZTERM_PANE` 探测，我们把**专属变量放在最前**，`TERM` 放最后。
理由是 `TERM` 经常被 tmux、ssh、或用户自己改成 `xterm-256color`；照它判定会把一堆
终端误认成 xterm，进而发出根本不生效的 OSC 50。专属变量（`KITTY_WINDOW_ID` 等）
只有那个终端才会设，可靠得多。

**做不到的时候给配置片段**（§6.10 `[MUST]`，文档特意点了「不要省」）：
改不了字体就把该改的配置替用户写好——kitty.conf / wezterm.lua / alacritty.toml /
WT 与 VS Code 的 settings.json。说不准格式的终端（Apple Terminal 走图形界面、
未知终端）**不给**片段：硬编一段假配置比不给更糟。

## 未完成项

M6 的验收是「在 kitty/alacritty/WT 三种终端下行为正确且降级合理」。
目前只做到「降级逻辑正确且可测」，三种终端下的**实际**行为仍未验证：

- [ ] kitty：开/不开 `allow_remote_control` 两种情况下 `set-font-size` 的表现
- [ ] xterm/urxvt：OSC 50 改字体族是否生效；空参数是否恢复默认
- [ ] Alacritty / Windows Terminal：确认确实无运行时能力，且生成的片段贴进去有效
- [ ] 退出与 panic 后字体确实被还原（`[MUST]`，改过字体才发恢复序列的逻辑已就位）

验证方式：在各终端里跑 `mj doctor` 对照报告，再跑一次真实的字体切换。
这几项要么由能接触到这些终端的人补，要么在 M7 之前找机会补。
在补完之前，**不得**把 §2.1 的表当成已验证事实。
